Act I · 02 / 13
为什么需要语义聚类？

传统方案的问题

❌ 关键词匹配

"牛批" 和 "牛逼" 零字符重叠，正则表达式永远匹配不到。弹幕变体是无底洞——观众每天都在发明新的规避词。

❌ 编辑距离

2 字符编辑距离 = 遍历全部 12k 唯一文本进行 pairwise 比较。O(n²) 复杂度，200 msg/s 下 CPU 直接打满。

✅ 语义 Embedding

"牛批""牛B""niubi""牛逼" 在 BGE 向量空间中彼此靠近。不需要知道变体长什么样——模型自己学会了"这些词意思一样"。

预处理 · Pipeline Steps 1-3
Act II · 06 / 13
每一步解决一类噪声

预处理管线 (上)

① 基础清洗
移除控制字符、统一全角半角、NFKC 归一化、1-128 字裁剪。过滤纯数字 >4 字但不杀短数字梗 (666/520)。
目的: 把输入格式统一，扔掉明显垃圾，不确定的留给后续步骤。
O(n)
② jieba 分词
variants.yaml 联动 jieba 词典: 规范词 + 变体词全部加入词典并 suggest_freq 调权。只替换完整词，不碰子串——
"东南亭子" 不会被误杀成 "东南难听子"。
目的: 子串扫描改词级替换，消灭"小满好棒"→"主播好棒"这类别名错位。防子串误匹配是核心价值。
O(n)
③ 变体归一
双层: 谐音字典精确匹配 ("牛批"→"牛逼") + 规范词自身拼音自动反向生成。
声调天然防误匹配——lai2le ≠ lai4le。
目的: SimHash 对音近字完全无效 (汉明距离 ~40)。字典 + 拼音双保险覆盖 80%+ 变体，无需手动维护 pinyin 配置。
O(n)
① 基础清洗 → ② jieba 分词 + 别名替换 → ③ 变体归一化 (字典 + 自动拼音 + 声调)

预处理 · Pipeline Steps 4-6
Act II · 07 / 13

预处理管线 (下)

④ 循环压缩
(.+?)\1{2,} 正则反向引用。容忍尾部残余，不再要求整除——"加油加油加油加"→"加油加"。
目的: 弹幕刷屏产生大量重复文本，压缩后减少 Embedding 计算量，同时保留语义信息。
O(n)
⑤ SimHash
≥8 字长文本 + 汉明距离 ≤2 → 高置信度自动合并。短文本 (<8 字) 仅写候选日志，不自动合并。
目的: 变体字典 + 拼音已覆盖 80%+，SimHash 降级为辅助角色，处理字典遗漏的长文本近重复。
O(1)
⑥ 精确去重
哈希表 O(1) 查找 + 频次累加。原始弹幕文本作为成员保留在 DedupStore._members 中。
目的: 精确去重是所有语义归一化的终点——经过前面 5 步后同义的文本落到同一 canonical，计数自然累积。成员保留实现内容穿透。
O(1)
管线设计原则

每一步解决一类特定的噪声，各司其职、互不重叠。顺序固定：清洗→别名→变体→压缩→SimHash→去重。精确去重在最后，计数自然累积。管道正向查缓存，命中则短路。

④ 循环节压缩 → ⑤ SimHash 辅助模糊 → ⑥ 精确去重 + 频次计数

缓存 · Cache
Act III · 08 / 13
Pipeline + Embedding 合二为一

统一缓存池

4 个预处理阶段 + Embedding 共用一份 LRU 缓存。
任一阶段命中 → 同时拿到 canonical + embedding → 下游全部短路。

为什么要合并？

之前 Pipeline 缓存和 Embedding 缓存各自独立——Pipeline 命中了还要再查一次 Embedding。合并后一次命中 = 全跳过。Embedding 是全程最贵操作 (首次 ~150ms)，统一缓存让它也参与短路。

聚类 · Clustering
Act III · 09 / 13
在线 + 防漂移 + 周期维护

在线聚类引擎

Leader-Follower 双阈值实时匹配 → 每 300 条周期维护 (合并 + 拆分) → 热点晋升

双阈值锚点

簇心 0.65: 控制聚合范围
锚点 0.75: 控制语义纯度
锚点不随时间漂移——用 top_examples 的 embedding 做固定参照

长度惩罚

"帅啊"(3字) ≠ "帅啊我看XXX也就那样了"(13字)，ratio≥4.3x → penalty=0.80，拒绝合并。BGE 共享前缀易误判→长度惩罚补偿。

内部一致性

簇内成员与 centroid 平均相似度 < 0.50 → 退化簇拒绝新成员。防止已经杂乱的簇继续吸收不相关内容。

周期合并

两簇 centroid 相似度 > 0.92 → 合并到较大簇。减少碎片化，让同类语义聚合在一起。

K-Means 拆分

内部方差 < 0.50 → k=2 拆分。手写极简 K-Means (~20行)，随机种子 → 分配 → 迭代收敛。

情感守卫

"周杰伦唱得好" vs "周杰伦唱的烂"——共享词 ≥2 且一方含正面词一方含负面词 → 拒绝合并。

Leader-Follower 实时 · 合并/拆分 300 条周期 · 热点 >100 晋升 · 10min TTL

为什么需要热点语义？

常规 40 槽位 LRU 淘汰——热门话题可能被新话题挤出。热点区独立存储，保证高频语义不被冲走。

10min 无更新自动淘汰——直播话题有生命周期，老热点过期自动释放，不再占用资源。

晋升前 centroid 重算 → 选中心文本为摘要 → 独立合并/拆分 → 情感守卫 → 10min TTL
